## 1. What is jarvis

`jarvis` is a personal CLI tool for Linux workstations, written in pure bash. It runs on Ubuntu. No package manager, no external runtime.

The **source** lives in modular files under `src/`. The **distributable** is a single concatenated file at `command/jarvis`, generated by `./build`. End users install `command/jarvis` — they never see the source split. The modular layout is for you, the developer, so you never have to scroll through a thousand-line file to add one command.

**Shell:** bash (not sh, not zsh)  
**Platform:** Ubuntu (some commands are Ubuntu/GNOME-only)

## 2. Project structure

```
jarvis-command/
├── src/
│   ├── core/
│   │   ├── header.sh       ← identity: script_name, version, small_desc
│   │   ├── globals.sh      ← color variables
│   │   ├── helpers.sh      ← logging, UI helpers, guards, banner, show_help
│   │   └── main.sh         ← dispatcher (the case statement — always last)
│   └── cmd/
│       ├── cmd_lights.sh
│       ├── cmd_lock.sh
│       ├── cmd_unlock.sh
│       ├── cmd_tree.sh
│       ├── cmd_observe_vault_log.sh
│       ├── cmd_attendance.sh
│       ├── cmd_nmhunter.sh
│       └── cmd_bkash.sh
│       └── cmd_yourfeature.sh  ← new commands go here
│
├── command/
│   └── jarvis              ← BUILD OUTPUT — generated by ./build, committed to git
│
├── build                   ← concatenates src/ → command/jarvis
├── dev                     ← runs directly from src/ (no build needed during dev)
├── installer.sh            ← installs command/jarvis to the system
├── generate-completions.py ← generates _jarvis Zsh completion from the schema
└── jarvis-schema.json      ← completion schema — edit when adding commands
```

### What each `src/` file does

| File                  | Purpose                                                                   |
| --------------------- | ------------------------------------------------------------------------- |
| `src/core/header.sh`  | `script_name`, `version`, `small_desc`. Edit version here.                |
| `src/core/globals.sh` | All color variables. See section 5.                                       |
| `src/core/helpers.sh` | Every reusable function: logging, dividers, guards, banners, `show_help`. |
| `src/core/main.sh`    | The dispatcher `case` statement. Edit when adding a new command.          |
| `src/cmd/cmd_*.sh`    | One file per command. Self-contained: helpers + banner + help + `cmd_*`.  |

## 3. The build system

Two scripts handle the development workflow:

### `./dev` — run directly from source

```bash
./dev lights on
./dev bkash cashout from 1000
./dev --help
```

Sources all `src/` files at runtime. **Use this while developing** — no build step needed, changes are reflected immediately.

### `./build` — produce the distributable

```bash
./build
```

Concatenates the source files in this exact order into `command/jarvis`:

```
src/core/header.sh
src/core/globals.sh
src/core/helpers.sh
src/cmd/*.sh        ← alphabetical (glob order)
src/core/main.sh
main "$@"           ← entrypoint call appended by build
```

Run `./build` when you are done developing and ready to commit or install. The generated `command/jarvis` is committed to git so that `installer.sh`'s online mode can download it directly from `raw.githubusercontent.com`.

> **Never edit `command/jarvis` directly.** It is overwritten every time `./build` runs. All edits go into `src/`.

## 4. Versioning

Version is stored in `src/core/header.sh`:

```bash
version="X.Y.Z"
```

Format is `MAJOR.MINOR.PATCH`. Bump rules:

| Change type                          | Bump  |
| :----------------------------------- | :---- |
| New command added                    | MINOR |
| Bug fix inside an existing command   | PATCH |
| Breaking change to existing behavior | MAJOR |

**Always bump the version when making any change.** Even a one-line fix gets a PATCH bump.

## 5. Color system

Defined in `src/core/globals.sh`. These are the only color variables that exist. Do not add new ones without a real reason.

```bash
RESET='\033[0m'
BOLD='\033[1m'
DIM='\033[2m'
ORANGE='\033[38;5;209m'        # salmon-orange, primary brand color
BORANGE='\033[1;38;5;209m'    # bold orange, used for emphasis
DIM_ORANGE='\033[2;38;5;209m' # dim orange, used for decorative/secondary
BWHITE='\033[1;37m'            # bold white, used for labels and step titles
BGREEN='\033[1;32m'            # bold green, used for success (✔)
RED='\033[0;31m'               # red, used for errors only
YELLOW='\033[1;33m'            # yellow, used for warnings only
BLUE='\033[94m'                # blue, used for [INFO] tag only
BBLUE='\033[1;94m'             # bold blue, available but currently unused
```

The **brand palette is orange**. Blue is reserved for `[INFO]` only. Green for success only. Red for errors only. Yellow for warnings only.

## 6. Logging & UI helpers

Defined in `src/core/helpers.sh`. These are the only helpers you use inside any command. Never `echo` raw ANSI codes directly.

### Color wrappers (for ASCII art / banners only)

```bash
log_clr_l1()  # ORANGE     — mid brightness lines
log_clr_l2()  # BORANGE    — top/bright lines
log_clr_l3()  # DIM_ORANGE — dim/fade lines
```

These three are exclusively for multi-line ASCII art rendering. Do not use them for regular output.

### Text wrappers

```bash
log_txt_nm()  # plain echo (no color)
log_txt_bd()  # bold text
log_txt_dm()  # dim text — used for secondary/hint text
```

### Semantic logging

```bash
log_info()   # [INFO] in blue  — informational, non-critical messages
log_ok()     # ✔ in green      — confirms something succeeded
log_warn()   # ⚠ in yellow    — something is off but not fatal
log_fail()   # [ERROR] in red  — prints to stderr and exits 1
log_label()  # ▸ in orange     — used in summary blocks (key: value lines)
```

**`log_fail` always exits.** Never call it if you want to continue execution.

### UI structure helpers

```bash
show_divider()   # long ━━━ line in dim orange — major section separator
divider_small()  # shorter ━━━ line — used when terminal is narrow
thin_div()       # short ── line in dim — used under step headers
```

### Interactive input helpers

```bash
prompt_line "Question text" "default value"
# Renders:  ?  Question text (default: value) ›
# Does NOT read input — just prints the prompt.
# Always follow with: read -r VARIABLE_NAME

step "1" "Step Title"
# Renders:  [1]  Step Title
# Used to visually number steps in interactive commands.
```

## 7. Guard / require helpers

Defined in `src/core/helpers.sh`. These check for a precondition and call `log_fail` (exit) if it's not met. Always call them at the top of a `cmd_*` function that needs them.

```bash
require_ubuntu
# Exits if not running on Ubuntu.

require_apt_package "cmd" "package-name"
# Exits if `cmd` is not in PATH.
# Second arg is the apt package name shown in the error message.

require_external_dependency "cmd" "hint message"
# Same as above but for non-apt software.
# Second arg is a freeform hint shown in the error.

require_openrgb
# Shorthand for require_external_dependency for openrgb specifically.

require_dbus
# Checks for dbus-send.

require_gnome_screensaver
# Checks GNOME screen saver is reachable via dbus.

require_gnome_lock
# require_dbus + require_gnome_screensaver combined.

require_file "/path/to/file" "Description of the file"
# Exits if the file does not exist.
```

## 8. Banner system

Defined in `src/core/helpers.sh`. Jarvis has a terminal-width-aware main banner:

```bash
show_banner()
# Detects terminal width via `tput cols`.
# If width < 60: renders show_banner_1 (compact ASCII art) + divider_small
# If width >= 60: renders show_banner_2 (full block ASCII art) + show_divider
# Always ends with a hint line.

refresh_banner()
# clear + show_banner. Used when a command wants to wipe the screen first.
```

**Feature-specific banners** are private functions inside each `src/cmd/cmd_*.sh` file. Each interactive command that has its own flow gets `_show_<feature>_banner()`. It always:

- Renders its own block ASCII art using `log_clr_l1/l2/l3`
- Ends with `log_txt_dm "  Feature Name · Part of ${script_name} v${version}"`
- Calls `show_divider`
- Prints a blank `echo`

The feature banner is called with `clear` before it inside the `cmd_*` function, not inside the banner function itself.

Non-interactive commands (like `lock`, `tree`, `power`) do not need their own banner.

## 9. Help system

There are two tiers of help.

### Tier 1 — global help

`show_help()` lives in `src/core/helpers.sh`. Called when the user runs `jarvis --help`. It lists all available commands. **Always update this list when adding a new command.**

```bash
show_help() {
    cat << EOF
${script_name} - ${small_desc}

Usage:
    ...

Available commands:
    lights, lock, unlock, observe, monitor,
    tree, power, attendance, nmhunter, bkash, version, help
    # ↑ add your new command name here

...
EOF
}
```

### Tier 2 — per-command help

Each command that has sub-options or interactive steps gets `_show_<command>_help()` inside its own `src/cmd/cmd_*.sh` file. It is private (underscore prefix). Called inside `cmd_<name>` when the user passes `-h` or `--help`.

Format uses a `cat << EOF` heredoc. Plain text, indented with spaces. Keep it short: usage, flags, brief description. That's it.

## 10. Command anatomy

Every `src/cmd/cmd_*.sh` file follows this internal layout, top to bottom:

```bash
# =================================================================================
# == My Feature ===================================================================
# =================================================================================

# 1. Private helpers (if needed)
_myfeature_do_something() {
    ...
}

# 2. Feature banner (only if the command is interactive)
_show_myfeature_banner() {
    echo
    # ASCII art using log_clr_l1 / log_clr_l2 / log_clr_l3
    echo
    log_txt_dm "  My Feature · Part of ${script_name} v${version}"
    echo
    show_divider
    echo
}

# 3. Per-command help
_show_myfeature_help() {
    cat << EOF

  One line description.

  Usage:
      ${script_name} mycommand
      ${script_name} mycommand [flags]

  Flags:
      -h, --help    Show this help

EOF
}

# 4. Public cmd function — the only thing the dispatcher calls
cmd_myfeature() {
    # Guards first
    require_ubuntu

    # Flag / subcommand handling
    case "${1:-}" in
        -h|--help|h|help)
            _show_myfeature_help
            return 0
            ;;
    esac

    # Local variable declarations
    local VAR_ONE VAR_TWO

    # If interactive: clear + banner
    clear
    _show_myfeature_banner

    # Steps (if interactive)
    step "1" "Step Title"
    thin_div
    prompt_line "Question?" "default"
    read -r INPUT
    INPUT="${INPUT:-default}"
    # validate
    log_ok "Confirmed → ${BOLD}${INPUT}${RESET}"
    echo

    # Do the actual work

    # Summary block (if applicable)
    show_divider
    echo
    log_label "Key  :  Value"
    echo
    show_divider
    echo
}
```

**Critical rules:**

- All variables inside `cmd_*` must be declared `local`. No globals leaking out.
- Use `return 0` to exit early from a function, never `exit 0`. (`exit` kills the whole process.)
- Guards always go first, before any other logic.

## 11. Dispatcher (main.sh)

`src/core/main.sh` contains the execution block. It is **always the last source file concatenated by `./build`**. Bash executes top to bottom — every function referenced here must be defined by the time this runs.

**Stage 1** — flags that exit before subcommand processing:

```bash
case "${1:-}" in
    v | -v | version | --version)
        show_version
        exit 0
        ;;
    h | -h | help | --help)
        show_help
        exit 0
        ;;
esac
```

**Stage 2** — subcommand dispatch:

```bash
SUBCOMMAND="${1:-}"
shift || true   # shift safely even if no args

case "$SUBCOMMAND" in
    mycommand | mc | myalias)
        cmd_myfeature "$@"
        exit 0
        ;;
    ...
    *)
        show_help
        exit 0
        ;;
esac
```

**Rules:**

- Every command gets `exit 0` after its `cmd_*` call, except `observe` which uses `tail -f` and blocks.
- The `*)` fallthrough always calls `show_help`. Never silently fail.
- `shift || true` is intentional — `set -e` would kill the script if `shift` is called with no args.
- Aliases go in the same `case` pattern: `lights | light | ram | rams | led | leds | lt | lts`.

## 12. Private functions — the `_` prefix convention

Any function that is **not meant to be called by the dispatcher** gets an underscore prefix. This is jarvis's convention for "private" or "module-internal".

```bash
_myfeature_do_something()   # private helper, only used inside cmd_myfeature
_show_myfeature_banner()    # private banner
_show_myfeature_help()      # private help
```

Public functions (called by the dispatcher or by other public functions) have no underscore:

```bash
cmd_myfeature()
show_banner()
log_info()
require_ubuntu()
```

Because all `src/` files are concatenated into a single flat namespace, there is no real encapsulation in bash. The underscore is a communication tool — it tells the next person "don't call this from the dispatcher, it's an implementation detail." Scope collisions are prevented by always prefixing private helpers with the feature name: `_attendance_build_row`, not `_build_row`.

## 13. Interactive input pattern

Every interactive prompt in jarvis follows this pattern:

```bash
prompt_line "Question text?" "default_value"
read -r RAW_INPUT
RAW_INPUT="${RAW_INPUT:-default_value}"

# Validate
[[ ! "$RAW_INPUT" =~ ^[0-9]+$ ]] && log_fail "Not a number."
(( RAW_INPUT < 1 || RAW_INPUT > 100 )) && log_fail "Out of range."

FINAL_VAR="$RAW_INPUT"
log_ok "Label → ${BOLD}${FINAL_VAR}${RESET}"
echo
```

**Validation rules:**

- Integer check: `[[ ! "$VAR" =~ ^[0-9]+$ ]]`
- 4-digit year: `[[ ! "$VAR" =~ ^[0-9]{4}$ ]]`
- Range check: `(( VAR < MIN || VAR > MAX ))`
- String from known set: `case`/`esac` that emits `""` on no match, then `[[ -z "$RESULT" ]]`

Always show `log_ok` after successful input. Always print a blank `echo` after `log_ok` to separate steps visually.

## 14. Naming rules

**Functions**

- Public (dispatcher-facing): no prefix — `cmd_myfeature`, `show_help`, `require_ubuntu`
- Private (internal helpers): `_` prefix — `_myfeature_do_thing`, `_show_myfeature_banner`
- Always scope private helpers with the feature name: `_attendance_build_row`, not `_build_row`

**Variables**

- All variables inside any `cmd_*` function must be `local` — no exceptions
- Global variables are only declared in `src/core/header.sh` and `src/core/globals.sh`
- Use `SCREAMING_SNAKE_CASE` for globals, `lower_snake_case` for locals

**Before naming anything, check `src/core/main.sh`.**  
The dispatcher is the canonical list of what already exists. If a name is already in a `case` pattern — as a command or an alias — don't reuse it.

## 15. How to add a new command — step by step

This is the checklist. Follow it in order.

### Step 1 — Plan

Answer these before writing code:

- What is the command name? Check `src/core/main.sh` for conflicts.
- Does it need Ubuntu? GNOME? An external tool?
- Is it interactive (prompts the user) or fire-and-forget (just does a thing)?
- What aliases make sense?

### Step 2 — Bump the version

In `src/core/header.sh`:

```bash
version="1.11.0"  →  version="1.12.0"   # new command = MINOR bump
```

### Step 3 — Create `src/cmd/cmd_myfeature.sh`

Create a new file. Follow the command anatomy from section 10 exactly. Everything for this command lives in this one file — private helpers, banner, help, and the public `cmd_*` function.

```bash
# src/cmd/cmd_myfeature.sh

# =================================================================================
# == My Feature ===================================================================
# =================================================================================

_myfeature_do_something() { ... }

_show_myfeature_banner() { ... }    # omit if not interactive

_show_myfeature_help() { ... }

cmd_myfeature() { ... }
```

### Step 4 — Update `show_help()` in `src/core/helpers.sh`

Add your command name to the `Available commands:` list.

### Step 5 — Add to the dispatcher in `src/core/main.sh`

```bash
mycommand | mc | myalias)
    cmd_myfeature "$@"
    exit 0
    ;;
```

Add it before the `*)` fallthrough.

### Step 6 — Update `jarvis-schema.json`

Add the new command so tab completions stay in sync. See `SCHEMA-GUIDE.md`. Then regenerate:

```bash
python3 generate-completions.py
```

### Step 7 — Test with `./dev`

```bash
./dev mycommand --help       # help flag works
./dev mycommand              # happy path works
./dev mycommand garbage      # bad input fails cleanly with log_fail
```

`./dev` runs directly from `src/` with no build step. Use it throughout development.

### Step 8 — Build and verify

```bash
./build
bash -n command/jarvis       # syntax check — must pass with no output
./command/jarvis mycommand   # smoke test against the built file
```

### Step 9 — Install and test

```bash
./installer.sh update
exec zsh
jarvis mycommand --help
```

## 16. What not to do

These are real mistakes that were caught. Don't repeat them.

**Don't edit `command/jarvis` directly.**  
It is overwritten every time `./build` runs. Always edit the relevant file in `src/`.

**Don't use `exit` inside a function to return early.**  
Use `return 0` or `return 1`. `exit` kills the entire process, not just the function.

**Don't define global variables inside `cmd_*` functions.**  
Always use `local`. Without it, variables leak into the global scope and can stomp on other commands.

**Don't redefine any function from the core files.**  
If your feature needs a divider, call `show_divider`. Don't make your own `divider()`.

**Don't use `echo -e` with a variable that might contain `-e` as content.**  
Use `printf` when piping test input. In production code, `echo -e` is fine for styled output but never for user-supplied data.

**Don't add `set -euo pipefail` inside functions or inside `src/cmd/` files.**  
It is set once by `./build` in the generated file header. Adding it again does nothing useful.

**Don't leave dead color variables.**  
If you add a color variable and never use it, remove it. `BBLUE` is currently defined but unused — it stays because it may be needed soon, but don't add more.

**Don't forget `shift || true` before the subcommand dispatcher in `main.sh`.**  
Without `|| true`, `set -e` will kill the script when there are no arguments.

**Don't put any executable code in `src/cmd/` files.**  
Only function definitions. No bare statements. The `main "$@"` entrypoint is appended by `./build` after everything is concatenated — putting runnable code in a module file would execute it at `source` time, which breaks `./dev`.
